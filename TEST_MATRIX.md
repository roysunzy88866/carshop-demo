## 测试维度(6 列)

| 列 | 含义 | 例子 |
|---|---|---|
| **Happy** | 主流程正常 | admin/admin123 登录成功 |
| **Boundary** | 边界值 | 0 数量、负价、超长字符串、空 list、极大 page |
| **Negative** | 错误码 / 错入参 | 缺字段、id 不存在、权限不足 |
| **Special** | 特殊字符 / 编码 | emoji、引号、HTML 标签、中文、空格 |
| **Concurrent** | 并发 / 竞态 | 同时下单、改价时下单、删时查 |
| **Security** | 安全 | XSS、SQL 注入、path-traversal、CSRF |

### 车机端缺的关键测试(优先级)

🔴 **高优先级**:
- **下单并发**:同设备发两个 `POST /orders` 并发,订单号会不会撞?
- **支付并发**:同一笔订单短时间内两次 mock_pay,会不会重复处理?

🟡 **中优先级**:
- 商品 `description` 超长(10000 字符)是否会拖崩列表渲染
- 大 page 号(page=99999)返回什么
- App 在弱网环境下下单(超时、断网)

⚪ **低优先级**:
- title 含 emoji / 中文 / 特殊字符的渲染

### 后台缺的关键测试(优先级)

🔴 **高优先级 - 安全**:
- **title XSS**:新建商品 title=`<script>alert(1)</script>`,后台列表 / 车机详情会否执行?
- **图片 path traversal**:上传文件名 `../../../etc/passwd.png`,会不会写出沙箱?
- **banner image_url SSRF**:`http://169.254.169.254/`(云元数据)能否被请求?
- **暴力破解**:连错 N 次密码后是否限制?

🔴 **高优先级 - 数据**:
- **删商品 + 进行中订单**:OrderItem 是否真的能查到(product_id NULL + snapshot 保留)
- **改价 + 并发下单**:用户看到 ¥99,提交时变 ¥199 怎么办?(应锁定下单价 = 看到的价 ←需要确认设计)

🟡 **中优先级 - 边界**:
- 100+ 商品的分页正确性
- 极长描述 / 标题
- 改 price=0 是否允许
